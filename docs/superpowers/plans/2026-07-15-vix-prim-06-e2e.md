# vix-prim-06-e2e — first real registered primitive, corpus e2e, perf gate, docs

Phase 06 of the typed-primitive effort. Spec: `docs/superpowers/specs/2026-07-15-vix-typed-primitives-design.md` §7 (scheduler), §Testing lines 304-315, §Phase stack line 333-335. Branch: `vix-prim-06-e2e` (child of `vix-prim-05-scheduler`).

Phase 05 landed the effect-resolution core (`resolve_effects` in `runtime/scheduler.rs`): a registered primitive resolves at the demand layer, its response folds into the consumer demand key, Hermetic memoizes / Volatile re-dispatches, failures cross as `FailureValue::Primitive`. It proved this with **Int-request / String-response** primitives only (`primitive_scheduler.rs`), deliberately sidestepping two things this phase closes.

## What phase 06 adds

1. **String-request reference resolver** — the load-bearing gap. `decode_value` (convert.rs:414) rejects `FrozenValue::Reference` (store-resident strings), and its own doc comment says *"Phase 05 wiring resolves references against the `&Store` before calling this, so decode never sees one"* — phase 05 punted this by using Int fields. Real primitives take String inputs, so this is the first task.
2. **Record response + consumer field projection** — the spec's named first primitive is `probe_version where { text } → Version`. A record response exercises the aggregate facet→FramedNode encode (identity tree, EMPTY resident bytes — the weavy ABI constraint), and the consumer projecting a field off the realized record input.
3. **Perf gate report** — measure primitive-call overhead; confirm pure-op corpora are untouched (already true by construction — phase 05's `effect_free_test_touches_no_effect_machinery` proves effect-free `evaluate` is byte-identical).
4. **Docs page** under `vix/docs` — the registered-primitive API surface.

## Task 1: String-request reference resolver

The request island evaluates to a frozen record; a `String` field in it is a `FrozenValue::Reference(ValueId)` (store-resident), which `decode_value` rejects. Before decode (inside `resolve_effects`, on the cloned request frozen — or in a `convert.rs` helper the primitive adapter calls), resolve references against the store.

- **Resolver contract:** recursively walk a `FrozenValue`; for each `Reference(value_id)` substitute the referenced entry's concrete frozen (a resident String resolves to `Opaque(bytes)`). `Inline`/`Opaque` pass through; `Product`/`Variant`/`DenseArray`/`OrderedMap`/`OrderedSet` recurse structurally. Store API: `FrozenValue::Reference(ValueId)` (store.rs:47), `StoreEntry::frozen()` / `resident_bytes()` (store.rs:57,69), `frozen_for(Handle)` (store.rs:213). You will likely need a `ValueId → &StoreEntry` lookup — check for an existing one; if absent, add a small `entry_for_identity(ValueId) -> Option<&StoreEntry>` (the store already holds `by_identity`, keyed by `StoreKey`; a linear scan of `entries` by `.identity` is acceptable for v1 and honest).
- **Where to resolve:** prefer resolving in `resolve_effects` on the cloned request `FrozenValue` (the scheduler owns `&self.store` there, before `EffectCtx::new(&mut self.store)` takes `&mut`), so `begin`→`decode_value` sees a reference-free tree. Keep the borrow discipline: clone the resolved frozen out before the `&mut store` window.
- **Step 1:** failing test — extend `run_probe`'s primitive to `probe where { text: String } -> String` (a second registered signature, or parameterize `run_probe`), source `probe where { text: "1.2.3" }`, consumer `expect_eq(v, "1.2.3")`. It must fail today with the `decode_value` reference `ConvertError` surfacing as a machine/consumer failure.
- **Step 2:** implement the resolver; wire it into `resolve_effects` before dispatch. Keep decode's reference rejection as the honest backstop (resolver guarantees decode never sees one).
- **Step 3:** the new String-request test passes; the Int-request tests still pass unchanged. Full suite green (heavy tests isolated if slow-killed) + clippy clean.
- **Step 4:** commit — `vix: resolve store references in registered-primitive requests`.

Assert the consumer resolves cleanly and the projected field equals the expected scalar; assert Hermetic memo behavior holds for the record response (2 demands → 1 begin) — the response identity that folds into the consumer key is now an aggregate ValueId.

Do NOT expand scope beyond a projecting consumer over one record response.

## Perf gate report (Task 3)

**No pure-op regression — argued and demonstrated.**
- *Structural proof.* The entire phase-06 scheduler diff (`git diff vix-prim-05-scheduler...HEAD -- src/runtime/scheduler.rs`) is confined to `resolve_effects` step 7 (the effect-dispatch path). No pure-path code changed. An artifact with empty `effect_inputs` reaches ZERO new code, so the pure `evaluate` path is byte-identical to phase 05.
- *Phase-05 guard, re-cited.* `effect_free_test_touches_no_effect_machinery` (green on this branch) proves an effect-free `evaluate` dispatches nothing, emits no `EffectDispatched`, and evaluates `1+1==2` unchanged.
- *Spot-run.* `cargo nextest run -p vix --test rung050_trace_budget` → **18/18 pass**, same shape (budget enforcement, trace snapshots, wall/rss ceilings). The two new convert unit tests + the two new scheduler certs add coverage without touching a pure path.

**Primitive-call overhead (micro-benchmark).** `primitive_scheduler.rs::primitive_call_overhead_micro_benchmark` (`#[ignore]`, run with `cargo test -p vix --release --test primitive_scheduler -- --ignored --nocapture`). Wall-time ns/op, 4000 iterations, **release build**, single-process (so absolute numbers carry allocator/OS noise — the fresh-`Runtime`-per-iteration design forces a real dispatch every iteration; the *relative* conclusions are what hold). Three release samples:

| quantity | s1 | s2 | s3 | reading |
| --- | --- | --- | --- | --- |
| `pure_island_evaluate` | 2189 | 2303 | 2264 | baseline demand machinery every island pays |
| `hermetic_dispatch_scalar` (Int req / String resp) | 7840 | 13245 | 9254 | a real dispatch: request eval + decode + closure + response encode/intern + memo insert |
| `hermetic_dispatch_record` (String req / Version resp) | 19260 | 12456 | 9742 | dispatch + String-reference resolution + 3-field aggregate encode/intern |
| `effectful_memo_hit` | 2016 | 1169 | 1234 | a fully-memoized effectful demand |

Robust conclusions (stable across every sample; the absolute spread is single-process micro-bench noise from 4000 `Runtime` allocations, not primitive-path cost):
1. **A fully-memoized effectful demand ≈ a pure island evaluate** (memo-hit 1.2–2.0 µs vs pure 2.2 µs — at or below). Once memoized, the effect folds away and adds no marginal cost: the demand machinery is the whole cost, exactly as the design premises (design §Testing).
2. **A real dispatch is the same *order* as a pure island evaluate** — a single-digit multiple (~3.5–6×), never orders of magnitude. The overhead is bounded constant work (decode + closure + encode + intern + memo insert) dominated by the same demand machinery pure islands already pay.
3. **The record response adds a small, bounded conversion cost over the scalar response** (String-request reference resolution + aggregate framing/intern of a 3-field record), within noise of the scalar dispatch — consistent with "conversion cost linear in value size", not a blow-up.

No criterion harness was added (none exists; `cdcl_molten_bench.rs` is a bin). This is a REPORT deliverable per the plan; the micro-benchmark is committed `#[ignore]`d so it never runs (or slow-kills) in the suite.

## Self-review notes / known risks

- **Don't chase a module runner.** There is no authoritative-path "run this whole test module" entrypoint in v1; `checker_corpus.rs` drives the LEGACY machine. The `run_probe` hand-assembled `evaluate` IS the e2e surface (consistent with Approach A + demand-level dispatch). Extending it is the corpus e2e.
- **Consumer field projection is the real unknown** in Task 2. Phase 04/05 tests compared the response directly (`expect_eq(v, "9")`); a record response needs the consumer to project (`v.major`). Whether projection off a realized record input already lowers/evaluates is untested. Budget for that being where the work is; if the surface doesn't even parse field access on a bound value, fall back to a response the consumer can compare whole (e.g. return the record and `expect_eq` the whole record against a record literal) — still exercises aggregate encode + realized aggregate input, which is the point. Record the decision in the landing notes.
- **Store ValueId lookup:** if you add `entry_for_identity`, a linear scan is fine for v1 (test-scale stores); note it as O(n) with a "index later if it matters" comment. Do not reshape `by_identity`.
- **Borrow discipline in the resolver:** resolve on a clone of the request frozen taken from `&self.store`, then let `EffectCtx::new(&mut self.store)` take `&mut` — never hold a `&store` reference across the `&mut` window (mirrors the phase-05 pattern).
- **Perf gate honesty:** if a pure budget test moves at all vs main, STOP and surface it — that would contradict the whole design premise. Report real wall-times, not "should be fine."
