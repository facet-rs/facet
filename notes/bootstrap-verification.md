Status: verification pass against `vix/docs/content-design/runtime-closures.md`, section
"The seed — and the people who already solved it are allies". Outcome up front: **the
chain, the two names, and the direction ("Guix furthest along, Nix behind") are all
correct.** The one number that's wrong is the seed size — **357 bytes is stale by about
three years**; the actual current x86 seed is 181 bytes, and the document should either
drop the number or update it and cite the date. The bigger correction is **Nix's status is
out of date in the other direction**: as of a nixpkgs PR merged 2026-01-28, x86_64-linux
and i686-linux stdenv bootstraps *no longer use a binary `bootstrap-tools` seed at all* —
"work under way to reduce it" undersells what already shipped. One claim in the "give
back" section is unsupported as written and should be struck or softened: reproducibility-
by-observation does not exist anywhere in this ecosystem today: it's a genuinely new
capability vix would be offering, not something the ecosystem is inching toward. The
parsing claim is well-supported. Claim 7 surfaces a real prior-art item (Software
Heritage + Guix) that the document should probably name explicitly rather than risk
looking unaware of it.

## What the document should say instead

Replacement text for the paragraph currently marked "from memory, must be checked":

> The **Bootstrappable Builds** community — principally Jeremiah Orians (`stage0`,
> `stage0-posix`) and Jan Nieuwenhuizen (GNU Mes) — has spent years on exactly this: a tiny
> seed (`hex0`, a few hundred bytes and shrinking — 181 bytes on x86 as of this writing,
> down from 357 in 2022; small enough to audit by eye at any point in that history), a
> chain up through `hex1`/`hex2`/`M0`/`cc_x86` to `M2-Planet`, then `mescc-tools` and GNU
> Mes, then `tcc`, then a ladder of GCC versions (`live-bootstrap`, maintained by fosslinux,
> assembles all of this into one buildable end-to-end sequence, though it still requires an
> external preparation step today). **Guix** shipped a full-source bootstrap on that
> foundation in 2023 — with one honest caveat: a 25 MiB statically-linked Guile still ships
> as the build driver, so it's "no unexplained binary in the compiler chain," not literally
> zero binaries anywhere. **Nix** carried a binary `bootstrap-tools` seed for its entire
> history; as of early 2026, nixpkgs eliminated it for `x86_64-linux` and `i686-linux`
> (bootstrapping instead from the same `hex0` seed), while `aarch64-linux` and Darwin still
> use the older binary seed. Guix remains further along, but the gap is closing from both
> sides.

Replacement for the "give back" bullets — the read-set claim needs the "this is new"
framing restored, since checking it against reproducible-builds.org and `guix challenge`
shows nothing like it exists yet:

> - **Their chain has no equivalent today of reproducibility by observation.**
>   Reproducibility in this ecosystem currently means determinism plus independent
>   rebuild-and-compare (`guix challenge`, reproducible-builds.org's own definition) — no
>   project records, at exec granularity, what a bootstrap step actually read. A witnessed
>   read-set and a receipt per exec would be a new capability for this community, not a
>   faster version of something they already have.
> - **`snark` is ours, and bootstrap work is drowning in parsing.** Confirmed directly in
>   the source: `mescc-tools` hand-rolls ELF emission (`blood-elf.c`, `hex2_linker.c`),
>   `mescc-tools-extra` hand-rolls `tar`/`gzip`/`bzip2`/`xz` extraction as four separate C
>   programs, and `M2-Planet` hand-rolls its C-subset reader. A declarative binary dialect
>   is exactly the tool this work is missing.
> - **The transparency log and attestation format** are precisely the artifacts a
>   full-source bootstrap wants to publish. Worth naming explicitly that this is *not* the
>   same thing Software Heritage's Guix integration already provides — that's
>   content-addressed *source archival* (SWHIDs), useful for "can I still fetch this
>   source" but silent on "what did this build step actually read."

## What I could not verify

- **Exact date/version at which the Full-Source Bootstrap became part of a numbered Guix
  release** (vs. just "on the rolling channel via `guix pull`"). The blog and mailing list
  patch series give 2023-04-26 for the merge to master/`core-updates`; I did not find a
  Guix release-notes page tying it to e.g. "Guix 1.4.1" specifically in the time available.
  Doesn't affect any claim above — "shipped" is accurate either way.
- **Whether Guix's own vendored `bootstrap-seeds` copy tracks upstream `oriansj/bootstrap-seeds`
  HEAD or a pinned older commit.** This would explain precisely why the Guix blog's 357-byte
  figure (April 2023) matches a pre-July-2022 upstream revision rather than the 256-byte
  figure that was already current upstream by then. I could not fetch Guix's own
  `gnu/packages/bootstrap.scm`/`commencement.scm` package definitions to find the exact
  pinned commit/hash (repeated fetches of Guix's own source tree came back empty or 404 from
  the mirrors and cgit endpoints I tried). The conclusion ("357 was stale when quoted, and is
  more stale now") holds regardless of which specific pinned commit Guix used, since it's
  derived straight from `oriansj/bootstrap-seeds`'s own commit history/dates.
- **Whether `ar` archive parsing is hand-rolled anywhere in stage0/M2-Planet/mescc-tools
  or genuinely deferred until real `binutils` is available.** I found no `ar`-handling
  source file in `mescc-tools`, `mescc-tools-extra`, or `M2-Planet`'s top-level listings,
  but I did not do a full recursive source grep (M2libc, `test/` fixtures, or later
  `live-bootstrap` steps might reference it). Recommend the document either drop `ar` from
  its list or have someone grep the full trees before publishing.
- **live-bootstrap's current 64-bit/aarch64 support breadth.** Search results gave
  conflicting signals — an older issue says the project is 32-bit-only "for now," while
  `M2-Planet` itself (which live-bootstrap depends on) explicitly supports amd64/aarch64/
  riscv32/riscv64 code generation. I did not resolve exactly how far live-bootstrap's own
  *integration* (as opposed to M2-Planet's compiler backend) has gotten on 64-bit targets
  as of today. Doesn't affect any claim in the table, which didn't depend on 64-bit status.
